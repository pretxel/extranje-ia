# extranjeria-agent-chat Specification

## Purpose

TBD — captures the agentic chat surface for Extranjería.ai: an OpenAI-backed tool-calling assistant that answers Spanish immigration questions strictly from the BOE/extranjería corpus, with verifiable source citations, plan-aware accounting, and feature-flag gating.

## Requirements

### Requirement: Agentic chat endpoint
The system SHALL expose `POST /api/agent` that accepts a `messages: UIMessage[]` payload and streams an assistant response generated by an OpenAI-backed tool-calling loop.

#### Scenario: Authenticated user sends a question
- **WHEN** an authenticated user with available quota POSTs `{ messages }` to `/api/agent`
- **THEN** the response status is 200, content type is the AI SDK UI message stream, and the assistant turn is generated by `gpt-4o` invoking tools as needed

#### Scenario: Unauthenticated request
- **WHEN** a request without a valid Clerk session hits `/api/agent`
- **THEN** the system responds with HTTP 401

#### Scenario: User over plan limit
- **WHEN** a free-plan user with `queriesUsed >= 5` POSTs to `/api/agent`
- **THEN** the system responds with HTTP 402 and body `{ "error": "limit_reached", "plan": "free" }` and does NOT call the model

### Requirement: Single-provider OpenAI lock
The agent route SHALL use OpenAI for both embeddings and chat completion regardless of `AI_EMBEDDING_PROVIDER`.

#### Scenario: Google provider env is set
- **WHEN** `AI_EMBEDDING_PROVIDER=google` and a request hits `/api/agent`
- **THEN** embeddings used by the agent's retrieval tool are produced by OpenAI (`text-embedding-3-small` or value of `OPENAI_AGENT_EMBEDDING_MODEL`) and the chat model is `gpt-4o` (or `OPENAI_AGENT_MODEL`)

#### Scenario: OPENAI_API_KEY missing
- **WHEN** `OPENAI_API_KEY` is unset and a request hits `/api/agent`
- **THEN** the system responds with HTTP 500 and an error message indicating the missing key, before invoking any model

### Requirement: Corpus search tool
The agent SHALL be able to call a `searchExtranjeriaCorpus` tool that returns top-k relevant chunks from the BOE/extranjería corpus with their source document title and URL.

#### Scenario: Agent calls search tool
- **WHEN** the agent invokes `searchExtranjeriaCorpus({ query, k })` during a turn
- **THEN** the tool returns up to `k` (default 5, max 10) chunks each containing `content`, `documentTitle`, `documentUrl`, and `score`, sourced from `findRelevantChunks`

#### Scenario: No relevant chunks found
- **WHEN** the search tool finds zero chunks above the relevance threshold
- **THEN** the tool returns `{ chunks: [] }` and the agent's final response states that no verified source was found rather than fabricating an answer

### Requirement: Recent document changes tool
The agent SHALL be able to call a `listRecentDocumentChanges` tool that returns documents whose content changed within a recent window.

#### Scenario: Listing changes in last N days
- **WHEN** the agent invokes `listRecentDocumentChanges({ sinceDays })` with `sinceDays` between 1 and 365
- **THEN** the tool returns up to 20 documents ordered by `updatedAt` descending, each with `id`, `title`, `url`, `updatedAt`, where `updatedAt >= now - sinceDays`

#### Scenario: Default window
- **WHEN** the agent invokes `listRecentDocumentChanges({})` without `sinceDays`
- **THEN** the tool defaults `sinceDays` to 30

#### Scenario: Out-of-range window
- **WHEN** the agent invokes the tool with `sinceDays <= 0` or `sinceDays > 365`
- **THEN** the tool returns a validation error and the agent surfaces it as a fallback message instead of crashing the stream

### Requirement: Document detail tool
The agent SHALL be able to call `fetchDocumentDetail({ documentId })` to retrieve full metadata for a single document for citation.

#### Scenario: Existing document
- **WHEN** the agent invokes `fetchDocumentDetail({ documentId })` with a valid id
- **THEN** the tool returns `{ id, title, url, updatedAt, summary }`

#### Scenario: Unknown document id
- **WHEN** the documentId does not exist
- **THEN** the tool returns `{ error: "not_found" }` and the agent does not cite that document

### Requirement: Source citation transparency
For every claim in its final answer, the agent SHALL cite a source URL that was returned by a tool call within the same turn, emitted as a `source-url` UI message stream part.

#### Scenario: Agent cites only tool-returned URLs
- **WHEN** the agent finishes a turn that called retrieval tools
- **THEN** every `source-url` part written to the stream has a `url` value that appears in the tool results of that turn; URLs not returned by tools are filtered out

#### Scenario: Disclaimer is present
- **WHEN** the agent emits a final answer
- **THEN** the answer includes the standard disclaimer that the response is informational guidance and not legal advice

### Requirement: Tool-loop bound
The agent loop SHALL stop after at most 5 model steps per user turn.

#### Scenario: Step cap reached
- **WHEN** the model has produced 5 steps in a single turn without finishing
- **THEN** the loop stops and the partial answer is streamed to the user with a note that the agent could not complete the reasoning

### Requirement: Plan accounting
A single user turn on `/api/agent` SHALL increment `User.queriesUsed` by exactly 1, regardless of how many tool calls the agent made.

#### Scenario: Multi-tool turn counts as one
- **WHEN** the agent calls 3 tools in one turn and finishes successfully
- **THEN** `User.queriesUsed` increments by 1 in `onFinish`

#### Scenario: Failed turn does not bill
- **WHEN** the model errors before `onFinish` runs
- **THEN** `User.queriesUsed` is not incremented

### Requirement: Auth and routing protection
The Clerk middleware SHALL protect `/api/agent(.*)` and `/dashboard/agent(.*)`.

#### Scenario: Anonymous user hits dashboard agent page
- **WHEN** an unauthenticated browser navigates to `/dashboard/agent`
- **THEN** Clerk redirects to the sign-in flow

### Requirement: Document change tracking
The `Document` model SHALL expose `updatedAt` and `contentHash`, and the ingest pipeline SHALL only update a document row when its scraped content hash differs from the stored hash.

#### Scenario: Re-ingesting unchanged content
- **WHEN** the ingest pipeline scrapes a URL whose stored `contentHash` equals `sha256(scrapedContent)`
- **THEN** no write occurs for that document and `updatedAt` does not change

#### Scenario: Ingesting changed content
- **WHEN** the ingest pipeline scrapes a URL whose stored `contentHash` differs from `sha256(scrapedContent)`
- **THEN** the document's `content` and `contentHash` are updated, `updatedAt` advances, and prior chunks + their vectors for that document are deleted and re-embedded

#### Scenario: First-time ingest
- **WHEN** the ingest pipeline scrapes a URL not present in `Document`
- **THEN** a row is inserted with `contentHash = sha256(content)` and `updatedAt = createdAt`

### Requirement: Feature flag gating
The agent surface SHALL be gated behind environment variable `AGENT_CHAT_ENABLED`.

#### Scenario: Flag disabled
- **WHEN** `AGENT_CHAT_ENABLED` is unset or not equal to `"true"`
- **THEN** `/api/agent` responds with HTTP 404 and `/dashboard/agent` is not linked from navigation

#### Scenario: Flag enabled
- **WHEN** `AGENT_CHAT_ENABLED="true"`
- **THEN** the route is reachable and the dashboard nav exposes a link to it
